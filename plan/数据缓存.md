 # 面向大规模 DNA/FASTA 数据集的流式制作与按需读取改造方案

  ## Summary

  目标是在不改变用户可见输入参数、数据划分规则和监督语义的前提下，把当前“整批读入内存”的 DNA
  数据流程改成“流式制作 + 磁盘索引 + 按需读取”，使 500GB 级数据集可以在 60GB 内存机器上完成制
  作和训练准备。

  一致性要求按以下标准锁定：

  - 相同参数、相同随机种子下，train/query/base 的样本内容与顺序必须与当前实现完全一致。
  - 生成出的真实距离矩阵和 KNN 排序必须与当前实现完全一致。
  - 本次只保证“数据与缓存一致”；训练阶段现有的随机性缺陷保持不动，因为当前代码本身并不满足“同
    seed 完全复现训练结果”。

  ## Key Changes

  ### 1. 数据制作改为流式双遍扫描，但输出内容保持完全一致

  在 utils/fasta.py 引入新的内部实现，保留 prepare_dna_dataset() 现有对外参数和返回字段不变。

  核心做法：

  - 把 parse_fasta() 改成迭代器接口 iter_fasta_records(path)，逐条产出 (header, sequence)，不
    再返回整表。
  - 保留一个兼容层 load_dna_fasta(path)，仍返回完整 seq_ids, seqs，仅供旧调用点或小数据场景使
    用；新制作流程不再依赖它。
  - 新增流式校验函数，逐条完成：
      - upper()
      - 非 ACGT 校验
      - 长度统计
      - 记录 header
  - 对 train_fasta + eval_fasta 分支采用“两遍 eval、一次 train”的策略：
      - 第一遍扫描 train_fasta，顺序写入 train_seq_list 的中间分片，并累计 train_ids/
        train_size/max_len。
      - 第一遍扫描 eval_fasta，只收集 eval_count 和 eval_ids，不保存全部序列。
      - 按当前实现完全相同的规则生成划分：
          - order = list(range(eval_count))
          - random.Random(seed).shuffle(order)
          - split_at = int(round(len(order) * query_ratio))
          - split_at = min(max(split_at, 1), len(order) - 1)
          - query_idx = order[:split_at]
          - base_idx = order[split_at:]
      - 第二遍扫描 eval_fasta，按原始记录索引判断样本归属，分别顺序写入 query/base 的中间分
        片，同时构建 query_ids/base_ids。
  - 对 train_fasta + query_fasta + base_fasta 分支分别单独流式扫描三个文件；任一时刻内存中只
    保留当前记录和小批量写缓冲。
  - metadata.json 字段保持现状不变，字段值按现有语义构造，尤其是：
      - train_ids/query_ids/base_ids 仍为完整列表
      - train_size/query_size/base_size 完全一致
      - max_sequence_length 取三部分全局最大值
      - split_seed/eval_query_ratio 原样保留

  ### 2. 数据集落盘格式升级为“主数据文件 + 偏移索引”，并提供兼容导出

  当前 pickle 列表格式无法支持按需读取，因此需要引入新的主存储格式；同时保留兼容能力，确保旧
  逻辑可被验证。

  每个 split 统一生成两类文件：

  - 主数据文件：data/<dataset>/<split>.seqbin
      - 连续写入 UTF-8/ASCII 的序列字节
      - 每条记录独立存储，不拼成 Python 列表
  - 偏移索引：data/<dataset>/<split>.idx.npy
      - shape 为 [N, 2]
      - 每行存 (offset, length)
      - dtype=np.int64

  实现细节：

  - 制作阶段以固定批大小积累少量序列后写入 seqbin，同步记录偏移；不在内存中保留整 split。
  - 为了与当前实现对齐，写入顺序必须严格等于旧实现中列表顺序。
  - metadata.json 增加最小必要字段：
      - storage_format: "seqbin_v1"
      - train_storage/query_storage/base_storage
  - 保留一个可选兼容导出入口，把 seqbin + idx 重新导出成当前的 train_seq_list/query_seq_list/
    base_seq_list pickle，用于：
      - 回归测试
      - 与旧实现做 bitwise 对照
  - 默认训练与推理走新格式；不再要求必须存在 pickle 列表。

  ### 3. 训练与推理读取改为 split 级按需访问，不再构造 self.lines

  在 main.py 和 dataset/datasets.py 重构数据入口，但不改变训练上层调用方式。

  新数据层设计：

  - 新增 IndexedSequenceStore
      - 持有 seqbin 文件句柄和 idx.npy 的 memmap
      - 提供 __len__ 和 get(i)，按 offset 读取第 i 条原始序列
  - 新增 CombinedSequenceStore
      - 把 train/query/base 三个 store 视为一个逻辑总序列空间
      - 全局索引顺序必须等于当前 ReadData_fromfile() 拼接顺序
      - 支持从全局索引映射回具体 split 与局部索引
  - DataHandler 不再构造 self.lines
      - 只持有 CombinedSequenceStore
      - self.ni 由三个 split 长度之和得到
      - generate_ids()、load_ids() 逻辑保持不变
  - 对 DNA 数据新增“按索引惰性编码”流程：
      - 由于字母表固定为 ACGT，不需要扫描全量数据构建词表
      - M 直接从 metadata["max_sequence_length"] 读取
      - C = 4
      - 训练、query、base 三个集合只保存样本索引，不保存全部 char_ids
  - 把 StringDataset 改成支持两种模式：
      - 旧模式：接收内存中的 sig 列表，保持向后兼容
      - 新模式：接收 sequence_store + sample_indices + encoder
  - 新编码器行为必须与当前 word2sig(..., allowed_chars="ACGT", fixed_alphabet="ACGT") 完全一
    致：
      - A/C/G/T -> 0/1/2/3
      - __getitem__ 时再构造 one-hot [4, M]
      - padding 与当前实现一致，超出部分不允许出现
  - infer_fasta_embeddings.py 不再读取全部参考序列求 max_len
      - 直接从 metadata.json 读取 max_sequence_length
      - 只有待推理的输入 FASTA 按需读取

  ### 4. 距离矩阵与 KNN 缓存改为分块生成、mmap 读取

  当前 all_pair_distance() 会把整个距离矩阵一次性构造成内存数组；这对大规模数据同样不可接受。
  需要把“生成”和“读取”都改为磁盘导向，同时保证结果与现有实现一致。

  生成阶段：

  - 新增分块版距离计算接口，输入为：
      - 查询样本索引范围
      - base/train 的可读 store
      - 距离类型和数据类型
  - 以“行块”为单位计算距离：
      - 对 train-train：按查询块 A_chunk x full_train 计算，再写入 train_dist.npy 对应切片
      - 对 query-base：按查询块 Q_chunk x full_base 计算，再写入 query_dist.npy 对应切片
  - 使用 numpy.lib.format.open_memmap() 预创建目标 .npy
      - train_dist.npy
      - query_dist.npy
  - 每个行块完成后立即在块内做 np.argsort，把对应 KNN 行写入：
      - train_knn.npy
      - query_knn.npy
  - 排序必须保持与当前 get_knn() 一致：
      - 逐行 np.argsort(dist_row)
      - 不引入近似算法、不截断 top-k

  读取阶段：

  - load_dist() 对已存在缓存统一改为 np.load(..., mmap_mode="r")
  - TripletString 直接接受 memmap 类型的 dist/knn
  - 其余训练与评估逻辑不变

  ### 5. 一致性与回归验证策略内建到实现计划中

  为了保证新方案和旧方案在相同参数下结果完全一致，实施时必须同步提供对照验证脚本。

  验证步骤：

  - 小规模 FASTA 数据上同时运行“旧制作流程”和“新制作流程”：
      - 导出新格式对应的兼容 pickle
      - 比较 train/query/base 三个列表逐项完全相等
      - 比较 metadata.json 关键字段完全相等
  - 对同一份数据，比较新旧 train_idx/query_idx/base_idx 完全相等。
  - 对同一份数据，比较新旧 train_dist/query_dist/train_knn/query_knn 完全相等。
  - 在已存在小样本数据集上跑一次训练前准备，确认：
      - StringDataset[i] 的 one-hot 张量与旧实现逐元素一致
      - infer_fasta_embeddings.py 生成的输入编码与旧实现一致
  - 对大样本场景做资源验证：
      - 制作阶段 RSS 不随数据总大小线性增长到全量
      - 读取阶段不存在三份 split 全部驻留内存的峰值

  ## Public Interfaces / Files

  保持不变：

  - main.py 现有 CLI 参数
  - prepare_dna_dataset() 的入参与返回 metadata 结构
  - 训练、评估、推理命令行使用方式
  - 距离矩阵和 KNN 的 .npy 文件语义

  新增但不强制用户显式操作：

  - data/<dataset>/train.seqbin, query.seqbin, base.seqbin
  - data/<dataset>/train.idx.npy, query.idx.npy, base.idx.npy
  - metadata.json 中的存储格式标记字段
  - 一个内部或脚本级“导出兼容 pickle”工具，用于回归验证

  ## Test Plan

  - train_fasta + eval_fasta 模式：
      - 相同 seed、相同 eval_query_ratio 下，新旧 train/query/base 内容和顺序完全一致
  - train_fasta + query_fasta + base_fasta 模式：
      - 三个 split 内容和顺序完全一致
  - 错误条件一致：
      - 非法碱基报错一致
      - eval 少于 2 条时报错一致
      - train_dist/query_dist/train_knn/query_knn 与旧实现完全一致
      - 若抽取任意若干索引，新旧 StringDataset.__getitem__() 输出完全相同
  - 资源测试：
      - 500GB 级输入制作时不出现整表驻留内存
      - 读取阶段不再通过 pickle.load(...).extend(...) 构造全量 self.lines

  ## Assumptions

  - 本次范围聚焦 DNA/FASTA 制作入口，以及 DNA 训练/推理阶段的非整批读取。
  - protein/traj 没有现成“制作数据集”入口，不纳入本次主改造；若需要，可后续复用同样的 seqbin
    + idx 思路扩展。
  - “训练结果完全一致”本次按你的选择解释为“数据与缓存一致”；当前训练代码自身的非确定性行为保
    持不动，不作为本次兼容基准的一部分。
  - 为实现真正的按需读取，允许替换当前 pickle 列表 作为主存储格式；但必须保留兼容导出能力，以
    便严格证明新旧数据完全一致。
